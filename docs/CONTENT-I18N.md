# Content & Indic i18n Guide

How Yazhi's copy is written, stored, and rendered — and how to bring the
site to **any Indic language** without touching component code.

---

## 1. The three display modes

Every visitor picks a language *and* a mode (navbar, top-right):

| Mode | What renders | Example (hero scroll cue) |
|---|---|---|
| **Indic** | The chosen Indic language only | வாங்க, கீழே போகலாம் |
| **ENG** | English only | come, scroll with us |
| **Mixed** | One flowing, code-switched sentence | வாங்க — scroll பண்ணலாம் |

The Indic language is chosen in the picker next to the mode pills
(தமிழ், हिन्दी, … — languages without translations are marked *soon* and
fall back to English). The mode pills relabel themselves in the chosen
language: தமிழ் / ENG / கலவை for Tamil, हिन्दी / ENG / मिश्रित for Hindi.

Preferences persist in `localStorage` (`yazhi-lang-mode`,
`yazhi-lang-indic`). The old three-way `yazhi-lang` value (`ta`/`en`/`both`)
is migrated automatically on first visit.

## 2. The `LText` shape — one string, many voices

All copy lives in `src/lib/content.ts` as `LText` objects
(defined in `src/lib/i18n.tsx`):

```ts
scrollCue: L(
  "come, scroll with us",                                  // en
  { ta: "வாங்க, கீழே போகலாம்", hi: "आइए, नीचे चलें" },        // indic variants
  { ta: "வாங்க — scroll பண்ணலாம்", hi: "आइए — scroll करते हैं" } // mixed variants
)
```

- `en` — conversational English. Required.
- `indic` — per-language variants keyed by ISO code. **`ta` is required**
  (Tamil is the primary voice); every other language is optional.
- `mixed` — the authored code-switched line, per language. Optional.

**Fallback rules** (implemented in `resolveIndic` / `resolveMixed`):

1. Indic mode, language not yet translated → English (with an honest
   `lang="en"` attribute, so assistive tech isn't lied to).
2. Mixed mode, no authored mixed line → the stacked bilingual rendering
   (Indic line + English line), exactly like the old BOTH mode.

So a partially-translated language degrades gracefully, and adding
translations is purely additive.

## 3. The conversational voice

The site talks *to* the visitor, like a friend showing them around.
Rules of thumb, in every language:

- **Second person, present tense.** "Ask it anything…", "உள்ளே வாருங்கள்",
  "आइए, शुरू करें" — invitations, not announcements.
- **Short sentences. Questions are welcome.** "New here?" / "புதியவரா?"
- **Concrete over abstract.** Not "sovereign multilingual inference" but
  "start a sentence in English and finish it in Tamil — it keeps up."
- **Warmth over grandeur** — but never paraphrase quoted literature. The
  Bharathiyar line is verbatim, always; only its *translation* and the
  conversational lead-in around it follow the visitor's language.
- Tamil copy uses polite plural imperatives (சொல்லுங்கள், வாருங்கள்);
  Hindi uses आप-forms (कीजिए, आइए). Match that register when adding
  a language.

## 4. Writing mixed lines (the English → Indic hand-off)

The mixed voice is how bilingual speakers actually talk — *Tanglish*,
*Hinglish*, and their siblings. These lines are **authored by hand**,
never generated by gluing the two translations together. What makes a
switch feel natural:

- **Switch at phrase boundaries**, not mid-word:
  "Ask in Tamil, follow up in Hindi — நடுவில் எந்த translation layer-ும் இல்லை."
- **Keep tech/brand terms in English** (model, weights, GitHub, Discord)
  and let the Indic language carry the verbs and feeling:
  "GitHub-ல போய் பாருங்க", "Star पண்ணுங்க" → no; "Star பண்ணுங்க" → yes.
- **Case suffixes attach to English words with a hyphen** in Tamil
  ("weights-ஐ", "machine-ல", "GitHub-இல்") and flow directly in Hindi
  ("weights download कीजिए").
- **The emotional beat lands in the Indic language.** Lead with English
  facts, land in the mother tongue:
  "It's AI that code-switches, நம்மைப் போலவே."
- Light colloquial forms are welcome in mixed lines (வாங்க, பண்ணுங்க,
  இருக்கு) even where the pure-Indic column stays standard.

At render time, `Bi` splits a mixed line into script runs
(`segmentMixed`) so each run carries a truthful `lang` attribute —
screen readers, hyphenation, and font rules all stay correct.

## 5. Adding a new Indic language — checklist

Everything is data; no component changes needed.

1. **Register it** in `INDIC_LANGUAGES` (`src/lib/i18n.tsx`) if it isn't
   already there: code, endonym, script, Unicode range, native label for
   the "mixed" pill. Leave `ready: false` while translating.
2. **Translate the copy** in `src/lib/content.ts`: add your language's
   key next to every `ta:`/`hi:` entry. Grep for `hi:` to enumerate the
   full set — Hindi is complete, so it doubles as the translation
   checklist.
3. **Author the mixed lines** — don't machine-translate them. Recruit a
   speaker who actually code-switches; read them aloud; if it sounds like
   a form letter, rewrite it.
4. **Transliterate the thinai names** (`THINAI[].name`) — they are Tamil
   literary terms; other languages carry them phonetically (मरुदम,
   कुरिंजी, …).
5. Flip `ready: true` — the language becomes selectable in the picker.
6. Have a native speaker review before shipping (the same editorial gate
   Tamil copy goes through).

Notes for specific scripts: Marathi shares the Devanagari range with
Hindi; Assamese shares the Bengali range — segmentation just works.
Urdu (Arabic script, RTL) needs bidi handling in `Bi` before it can be
registered; do that work first.

## 6. Where things live

| File | Role |
|---|---|
| `src/lib/i18n.tsx` | Language registry, modes, `LText` type, resolution + segmentation, `LangProvider`, `useLang`, `useText` |
| `src/lib/content.ts` | **All site copy** — the only file translators touch |
| `src/components/ui/Bi.tsx` | The multilingual text primitive (renders `LText` per mode) |
| `src/components/ui/LangToggle.tsx` | Language picker + mode pills |

`Bi` still accepts raw `ta`/`en` props for composed one-off strings, but
those only know Tamil — prefer `text={someLText}` everywhere new.
